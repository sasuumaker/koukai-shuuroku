# 非エンジニア向け個人開発基礎講座 - カリキュラム詳細

## 前提

- 1回30分、超初心者向け、スローペース
- 「ターミナルって何？」レベルの人が対象
- 回数は何回になってもOK
- コードは見ない。見るのはenvファイルのコピペだけ

---

## 第1回: この講座で何ができるようになるか

### 学ぶこと
- 完成品（TODOアプリ）のデモを見る
- 「コードを書かずにアプリを作る」とはどういうことか
- 使うツールの全体像（それぞれの役割）

### ステップ
1. 完成したTODOアプリをブラウザで見せる
   - タスクの追加・編集・削除・完了を実演
   - 「これを皆さんも作ります」
2. ツールの関係図を見せる
   - Cursor = 作業場（工房）
   - Claude Code = 職人（あなたの代わりにコードを書く）
   - GitHub = 倉庫（コードを保管する場所）
   - Vercel = 展示場（ネットに公開する場所）
   - Supabase = データベース（データを保存する場所）
3. 「コードは見ません」宣言
   - 皆さんがやるのは「日本語で指示を出す」こと
   - 唯一触るコードっぽいもの = envファイル（設定値のコピペだけ）
4. 必要なアカウント一覧を伝える（次回までに作らなくてもOK、講座内でやる）
   - Anthropic（Claude）アカウント
   - GitHubアカウント
   - Vercelアカウント
   - Supabaseアカウント

### ゴール状態
- 「自分にもできそう」と思ってもらう
- 全体の流れをぼんやりイメージできている

7. 画面の説明（必要最低限だけ）

   - 下 = ターミナル（次回ここを使う）

## 第3回: ターミナルに触れてみよう

### 学ぶこと
- ターミナルとは何か
- ターミナルの開き方
- 最初のコマンド体験

### ステップ
1. ターミナルとは何か
   - 「文字でパソコンに指示を出す窓」
   - マウスでクリックする代わりに、文字で命令する
   - 怖くない。間違えても壊れない
2. Cursorでターミナルを開く
   - メニュー: View → Terminal
   - またはショートカット: Ctrl + `（バッククォート）
3. 最初のコマンドを打ってみる
   - `pwd` → 今いる場所が表示される
   - `ls` → ファイル一覧が表示される
   - 「エンターキーを押すと実行される」を体感
4. 間違ったコマンドを打ってみる
   - `hello` → エラーが出る → 「エラーが出ても大丈夫」を体感
5. フォルダを作ってみる
   - `mkdir my-first-app` → フォルダができた
   - `cd my-first-app` → フォルダの中に入った
   - Cursorの左サイドバーにもフォルダが見える（連動している）

### ゴール状態
- ターミナルを自分で開ける
- コマンドを打って実行できる
- 「文字でPCに指示を出す」感覚を掴んでいる

## 第8回: Supabaseプロジェクトを作ろう

### 学ぶこと
- Supabaseとは何か
- プロジェクト作成
- API KeyとURLの場所を確認

### ステップ
1. Supabaseとは何かを説明
   - 「データを保存する場所」
   - TODOアプリなら「タスクの一覧」を保存する
   - ユーザー認証（ログイン機能）もやってくれる
2. supabase.comにアクセス
3. 「Start your project」からアカウント作成（GitHub連携推奨）
4. 新しいプロジェクトを作成

5. プロジェクトが作成されるのを待つ（1-2分）
6. API Keyの場所を確認
   - Settings → API → Project URL をコピー
   - Settings → API → anon public key をコピー
   - 「この2つを後でenvファイルに貼り付けます」
   - どこかにメモしておく（メモ帳でOK）

### ゴール状態

## 第9回: 開発プロジェクトを作ろう（envファイル設定）

### 学ぶこと
- プロジェクトフォルダの作り方
- envファイルとは何か
- Supabaseの値をenvファイルに設定

### ステップ
1. プロジェクト用フォルダを作る
   - ターミナルで: `mkdir ~/my-todo-app && cd ~/my-todo-app`
   - Cursorで「Open Folder」→ このフォルダを開く
2. Claude Codeを起動
   - ターミナルで: `claude`
3. Claude Codeに「Next.jsプロジェクトを作って」と指示
   - Claude Codeが `npx create-next-app` 等を実行
   - 許可を求められたら「Allow」
   - 完了を待つ
4. envファイルとは何かを説明
   - 「アプリの設定値を書くファイル」
   - 「パスワードみたいな大事な情報を入れる場所」
   - 「これだけは自分で値を入れる必要がある」
5. envファイルを作成
   - Claude Codeに「.env.localファイルを作って。中身はSupabaseのURLとKeyを入れる場所を用意して」と指示
   - Claude Codeがテンプレートを作る
6. Supabaseの値をコピペ
   - 前回メモしたProject URLとanon keyを貼り付ける
   - Cursorの画面で.env.localファイルを開いて、値を入れる
   - **ここが唯一「コードっぽいもの」を触る瞬間**

### ゴール状態
- Next.jsプロジェクトが作成されている
- .env.localにSupabaseの接続情報が入っている

## 第15回: PRで修正を体験しよう

### 学ぶこと
- PR（プルリクエスト）とは何か
- Claude Code Webで修正指示を出す
- プレビューで確認してマージする

### ステップ
1. PRとは何かを説明
   - 「変更の提案書」
   - いきなり本番を変えるのではなく、「こう変えたいんですけどいいですか？」と提案する仕組み
   - 確認してOKなら「マージ」で反映
2. Claude Code Webで修正指示を出す
   - 例: 「TODOアプリのタイトルを"My TODO"から"○○のTODOリスト"に変更して」
   - Claude Code Webが作業を開始
3. PRが作成される
   - GitHubにPRが作られたことを確認
   - 「何が変わったか」の差分を見る（赤と緑の表示）
   - 「赤が消えた部分、緑が追加された部分です」
4. Vercelプレビューを確認
   - PRに自動でプレビューURLが付く
   - プレビューURLを開いて、変更が反映されているか確認
   - 「本番にはまだ反映されていません。プレビューだけです」
5. マージする
   - GitHubでPRを開く
   - 「Merge pull request」ボタンをクリック
   - 本番URLを確認 → 変更が反映されている

### ゴール状態
- PRを作成→プレビュー確認→マージ→本番反映の流れを体験
- 「修正はこのサイクルでやる」を理解

## 第17回: 振り返りとまとめ

### 学ぶこと
- 全体の流れの復習
- 各ツールの役割の再確認
- 「その他講座」「部活」への導線

### ステップ
1. 全体の流れを振り返る
   - 環境構築 → 開発 → デプロイ → 運用修正
   - 「皆さんはもうアプリを公開できる人です」
2. セキュリティモデルの復習
   - ローカルClaude Code = 管理者（構築用）
   - Claude Code Web = PR経由（運用・委任用）
   - envファイルの大切さ（絶対に人に見せない）
3. 次のステップの紹介
   - 「その他講座」: 別のアプリを作る（note自動投稿、SEOツール等）
   - 「部活」: 自分の好きなプロダクトを1ヶ月で作って公開
4. 困ったときの対処法
   - Claude Codeに聞く（エラーメッセージを貼り付けるだけ）
   - Supabaseの自動停止（7日放置）→ ダッシュボードから再開
5. Q&A / 感想共有

### ゴール状態
- 基礎講座の全スキルを習得
- 次のアクション（その他講座 or 部活）が明確
